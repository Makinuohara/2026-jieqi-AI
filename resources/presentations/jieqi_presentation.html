七模块分层架构

项目采用 Maven 多模块 组织，严格遵循分层解耦原则。每个模块通过接口定义契约，底层模块不感知上层调用者，实现了高内聚、低耦合的设计目标。

  • 不可变状态设计：GameState、Board、Move 均为不可变记录类。通过创建新实例而非修改推进状态，从根本上消除并发隐患。
  • 信息隐藏原则：AI 仅接收 PlayerView（公开棋盘副本），永不获取暗棋真实身份。联网模式下，服务端不向客户端暴露对方暗棋信息。
  • 可替换接口设计：Agent、Evaluator、GameEngine 均为接口。三种 AI 可热替换，估值函数与规则引擎均可独立演进。

Expectiminimax 深度搜索

区别于传统 Minimax，揭棋特有的暗棋翻牌引入了随机性——翻牌走法的结果不确定。Expectiminimax 在博弈树中引入机会节点，对每种可能的翻牌结果按其概率计算期望值，从而做出最优决策。

  • Alpha-Beta 剪枝：在 MAX/MIN 层剪枝减少搜索空间，提升搜索深度。机会节点不可剪枝但可用概率权重优化遍历顺序。
  • 置换表 (Transposition Table)：200K 条目缓存，相同局面避免重复搜索。Key 由棋子位置、暗棋状态、历史翻牌信息哈希生成。
  • 静态搜索 (Quiescence Search)：到达叶子节点后继续搜索吃子走法，避免"地平线效应"导致的估值误判。

项目背景

什么是揭棋？

揭棋是中国象棋的变体，引入战争迷雾机制。除将/帅外所有棋子初始面朝下隐藏，首次移动或吃子时翻面揭示真实身份。

  • 暗棋机制：隐藏棋子按其起始位置的"虚拟类型"移动，翻面后揭示从暗棋池随机抽取的真实类型。
  • 翻面后规则变化：士/仕、象/相翻面后可过河、出宫，不再受限。
  • 消灭将/帅即获胜：规则允许"送将"（不考虑逃将义务），将军不需躲避。

核心算法

Expectiminimax 搜索

在揭棋中，翻牌走法具有随机性。传统 Minimax 无法处理随机节点。Expectiminimax 在博弈树中引入机会节点，以概率加权期望值指导决策。

  • Alpha-Beta 剪枝：在确定层剪枝，机会节点按期望值传播。
  • 置换表：200K 条目的 HashMap，Key 由 Zobrist 哈希生成。
  • 自适应深度：将军时加深至 5 层，可杀棋时 6+ 层，残局自动加深。
  • 安全过滤：搜索前先排除会导致己方被将的走法。
// 机会节点：枚举所有可能的翻牌结果 for (PieceType revealedType : possibleTypes) { double prob = pool.probabilityOf(revealedType); double newStateValue = expectiminimax( stateAfterRevealing(revealedType), depth - 1, alpha, beta, !isMaxPlayer ); expectedValue += prob * newStateValue; // 机会节点不剪枝，但要累加概率 }
规则实现

完备的揭棋规则引擎

StandardGameEngine 近 550 行代码，覆盖揭棋所有规则细节，通过 89 项自动化测试验证：

  • 7种棋子全部走法：车直行、马走日（蹩脚检测）、炮翻山（炮架检测）、兵/卒过河前后差异。
  • 边界与限制：飞将检测（将帅相对）、连将 6 次判负、兵卒连捉限制。
  • 终局判定：将帅被吃、无子可动（困毙判负）、80 回合无吃子判和。

联网架构

实时联网对弈系统

基于 WebSocket 协议的全双工实时通信，支持多房间并发对弈：

  • 玩家大厅：在线玩家列表、快速匹配队列、邀请/接受/拒绝机制。
  • 先后手协商：双方 Ready 后随机分配先手红方。
  • 服务端权威仲裁：走法合法性校验、暗棋翻牌随机抽取（仅服务器决定）、终局判定。
  • 超时与断线：60秒走棋时限 + 5秒网络宽限。心跳检测断线，超时判负。
