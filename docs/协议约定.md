# 通信协议约定

## 规定优先级

规则冲突时按以下顺序处理：

1. `问题回答.txt`
2. `2026大作业公共接口.docx`
3. `2026大作业——揭棋.docx`
4. Unveil PDF 中不冲突的参考实现

## 基础通信

- 使用 WebSocket 传输 UTF-8 编码的 JSON 消息。
- 服务器默认监听 `8887` 端口。
- 每个文本帧必须严格小于 1024 字节。
- 未知的可选字段应兼容读取。
- 收到未知 `messageType` 时不能导致服务器崩溃。
- 首版使用临时用户和内存匹配，不依赖数据库。
- 首版只保证同时运行一间对局室。

## 时间规则

- 联网对局以服务器时间为准。
- 每步思考时间为 60 秒。
- 额外提供 5 秒网络传输容差。
- 超时由服务器判定，客户端倒计时只用于显示。

## 棋局规则

- 将帅初始明置，其他棋子初始为暗子。
- 不允许原地翻子。
- 暗子只在第一次合法移动或吃子后翻开。
- 暗子移动时使用所在初始位置对应的虚拟棋种规则。
- 翻出类型由服务器从该方剩余暗子类型池中随机抽取。
- 连续 80 个半回合没有吃子时判和。
- 长将和长捉必须连续发生，在同方第 6 次时触发判定。
- 普通长将或长捉判违规方负。
- 兵卒长捉任意棋子判和，兵卒长将仍判负。
- 玩家可以不应将，对方随后可以直接吃将帅获胜。

## 公开信息与暗子脱敏

`gameStart.initialBoard` 只能包含公开信息。将帅发送真实明子类型，其他暗子使用
所在位置对应的虚拟类型，不能提前泄露真实身份。

暗子被吃时：

- 吃子方收到真实的 `capturedType`。
- 被吃方收到字符串 `"NULL"`。
- 观战端不能收到真实类型。
- 移动暗子的 `flipResult` 按公共接口向双方广播。

`capturedType` 是向后兼容的可选字段。旧客户端忽略未知字段后仍应继续工作。

## 服务端裁决

- 客户端只提交走法，不能提交或指定翻子结果。
- 服务器负责走法校验、随机翻子、计时和终局判断。
- 非法走法只回复发送方。
- 合法走法结果广播给对局双方。
- 联网客户端和服务器不能共享同一个可变 `Board` 对象。

## 棋谱

棋谱应记录：

- 每一步的起点和终点。
- 服务端收到并处理走法的时间。
- 暗子实际翻出的类型。
- 被吃棋子的实际信息及发送给各接收方的脱敏结果。
- 最终胜负、和棋、认输或超时原因。

复盘时必须读取已经保存的随机翻子结果，不能重新生成。
